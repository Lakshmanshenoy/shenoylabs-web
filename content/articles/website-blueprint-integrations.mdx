## Atmospheric Entry

Integrations are where elegant architecture quietly breaks.

The app can remain static-first and calm, but every external service introduces a moving contract. The work is not merely wiring APIs; the work is preserving trust boundaries across time.

<ConceptReference title="Integration work is continuity work" concepts={["Trust Boundaries", "Failure Isolation", "Operational Memory"]}>
Each token, endpoint, and callback is a future maintenance obligation. This investigation exists to keep those obligations explicit and auditable.
</ConceptReference>

## Cognitive Orientation

This blueprint is organized around four layers:

1. Service purpose and boundary.
2. Required environment variables and exposure class.
3. Failure modes and visible symptoms.
4. Stewardship actions that preserve long-term reliability.

## Layer 1: Integration Surfaces

### TinaCMS

- Purpose: structured content authoring.
- Boundary: authoring-time and build-time workflows.
- Core vars: `TINA_TOKEN` (server), `NEXT_PUBLIC_TINA_CLIENT_ID` (public).

### Upstash

- Purpose: low-volume state (rate limit buckets, queues, ephemeral keys).
- Boundary: server-side APIs.
- Core vars: `KV_REST_API_URL`, `KV_REST_API_TOKEN`, `KV_REST_API_READ_ONLY_TOKEN`.

### Turnstile

- Purpose: form abuse mitigation.
- Boundary: client challenge + server verification.
- Core vars: `NEXT_PUBLIC_TURNSTILE_SITE_KEY`, `TURNSTILE_SECRET_KEY`.

### Email delivery

- Purpose: transactional contact flow.
- Boundary: server-side send actions.
- Core vars: `RESEND_API_KEY`, `CONTACT_FROM_EMAIL`, `CONTACT_TO_EMAIL`.

### GitHub and Vercel automation

- Purpose: repository automation and deployment configuration.
- Boundary: CI/server scripts only.
- Core vars: `GITHUB_TOKEN`, `GITHUB_REPOSITORY`, `GITHUB_BASE_BRANCH`, `VERCEL_TOKEN`, `PROJECT_ID`.

<ResearchCallout title="Boundary principle">
If an environment variable can be public, mark it `NEXT_PUBLIC_*`. Everything else remains server-only and should be treated as revocable, rotated infrastructure secrets.
</ResearchCallout>

## Layer 2: Failure Modes That Matter

Instead of listing every possible error, isolate failures by reader impact.

<TechnicalDepth title="Impact-ordered failure model">
- **Critical visibility failures:** DNS misconfiguration, bad domain mappings, or broken deploy routing.
- **High-friction transactional failures:** Turnstile validation errors, email provider outages.
- **Workflow degradation:** Upstash throttling, GitHub API quotas, CI build bottlenecks.
- **Editorial slowdown:** CMS token expiry, schema drift, non-critical preview issues.
</TechnicalDepth>

### First-response sequence

1. Identify boundary: client, route handler, external API, or deploy platform.
2. Verify env-var presence and scope.
3. Reproduce on staging with sanitized payload.
4. Inspect provider dashboard for quota or auth anomalies.
5. Record incident notes in runbook language.

## Layer 3: Semantic Continuity Across Systems

Phase 3 and beyond require integrations to feel interconnected without adding visible complexity.

Operationally that means:

- shared vocabulary for incidents
- consistent token rotation cadence
- cross-reference between integration docs and runbooks
- canonical investigations for recurring failure patterns

## Layer 4: Stewardship Discipline (Phase 5–6)

To keep this blueprint relevant over years:

- rotate high-risk secrets every 60 to 90 days
- review public vs server exposure quarterly
- run scheduled E2E checks for contact and consent flows
- keep failure notes close to code and runbooks
- reject integrations that add complexity without conceptual benefit

This is where technical rigor becomes editorial integrity.

## Reflective Expansion

A mature integration blueprint does not try to predict everything. It preserves orientation when systems drift.

As ShenoyLabs evolves, this investigation should deepen through revisions, not sprawl through endless one-off notes.

If the platform remains calm during failure, continuity is working.
